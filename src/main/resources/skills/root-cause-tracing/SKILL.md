# root-cause-tracing Skill

## 核心原则

**修根因,不修症状。** 报错抛出的那一行,常常只是「坏数据流到这里终于崩了」的地方,不是它变坏的地方。

## 方法

1. **抓住表象**:精确记录出错现象(异常类型 + 栈 + 触发输入)。
2. **反向回溯数据/调用**:从出错点往上游问——「这个坏值/坏状态是谁给的?」顺着调用链、赋值链、事件链往回走。
3. **连问 5 个为什么**(5 Whys):
   - 为什么崩?→ 值是 null。
   - 为什么是 null?→ 上游没初始化。
   - 为什么没初始化?→ 某分支漏了。
   - 为什么漏?→ 新增分支时没覆盖……直到触底(能改一处就根治的那层)。
4. **确认根因**:在根因处能解释**全部**表象;修这一处,所有相关症状一起消失。
5. **最小修复 + 回归测试**:在根因层修,并补一个能复现原 bug 的测试(见 [[test-driven-development]])。

## 判断「到底了没」

- 还能继续问「那为什么会这样」且答案落在别处 → 还没到根因。
- 修这里只是让某个症状不出现,但同类问题换个入口还会来 → 修的是症状。
- 找到「改这一处,一类问题整体消失」的层 → 到根因了。

## 与 wraith 的衔接

- 配合 [[systematic-debugging]]:系统排查负责「稳定复现 + 缩小范围」,本技能负责「范围内挖到底」。
- 跨进程 bug(桌面↔app-server)顺 RPC 链两侧一起追;别只在一端看。

## 反模式

- ❌ 在报错行加 null 判断/try-catch 掩盖,不问值为何是 null。
- ❌ 追一层就停,改了个中间症状。
- ❌ 找到根因却「顺便」大重构,混淆修复与改造。
