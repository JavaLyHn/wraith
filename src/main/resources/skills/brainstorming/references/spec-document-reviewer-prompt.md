# Spec 文档评审 prompt 模板

派发「spec 文档评审」子 agent 时用这个模板。

**目的:** 核实 spec 是否完整、一致、可进入实现规划。

**何时派发:** spec 文档已写到 docs/superpowers/specs/ 之后。

```
子 agent(general-purpose):
  description: "评审 spec 文档"
  prompt: |
    你是一名 spec 文档评审员。核实这份 spec 是否完整、可进入规划。

    **待评审 spec:** [SPEC_FILE_PATH]

    ## 检查什么

    | 类别 | 看什么 |
    |------|--------|
    | 完整性 | TODO、占位符、"TBD"、未完成的章节 |
    | 一致性 | 内部矛盾、互相冲突的需求 |
    | 清晰度 | 需求是否歧义到会让人「造错东西」 |
    | 范围 | 是否聚焦到单份计划可覆盖——没有横跨多个独立子系统 |
    | YAGNI | 没被要求的功能、过度工程 |

    ## 校准(重要)

    **只标记那些会在实现规划阶段造成真实问题的项。**
    缺一个章节、一处矛盾、或一条歧义到能被两种解读的需求——这些是问题。
    措辞的小改进、风格偏好、「某节没别节详细」——这些不是。

    除非存在会导致计划有缺陷的严重缺口,否则批准。

    ## 输出格式

    ## Spec 评审

    **状态:** 批准 | 发现问题

    **问题(若有):**
    - [章节 X]:[具体问题] -[为何对规划重要]

    **建议(仅供参考,不阻断批准):**
    - [改进建议]
```

**评审员返回:** 状态、问题(若有)、建议
