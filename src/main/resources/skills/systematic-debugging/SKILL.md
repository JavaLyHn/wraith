# 系统化调试(Systematic Debugging)

## 概述

随机修改浪费时间、制造新 bug。快速补丁掩盖底层问题。

**核心原则:动手修之前,永远先找根因。修症状 = 失败。**

**违反本流程的字面,就是违反调试的精神。**

## 铁律

```
没有先做根因调查,就没有修复
```

如果你还没完成阶段一,你就不能提出修复。

## 何时用

用于任何技术问题:测试失败、生产 bug、非预期行为、性能问题、构建失败、集成问题。

**尤其在以下情况用:**
- 时间压力下(紧急时最容易忍不住瞎猜)
- 「就一个快速修复」看起来很显然
- 你已经试了好几个修复
- 上一个修复没用
- 你没完全理解这个问题

**别在这些时候跳过:**
- 问题看着简单(简单 bug 也有根因)
- 你在赶时间(赶 = 保证返工)
- 老板要你现在就修好(系统化比乱试快)

## 四个阶段

你必须完成每个阶段才能进下一个。

### 阶段一:根因调查

**在尝试任何修复之前:**

1. **仔细读错误信息**
   - 别跳过错误或警告
   - 它们常含确切的解法
   - 完整读栈追踪
   - 记下行号、文件路径、错误码

2. **稳定复现**
   - 你能可靠触发它吗?确切步骤是什么?每次都发生吗?
   - 不能复现 → 收集更多数据,别猜

3. **检查近期改动**
   - 什么改动可能导致这个?git diff、近期提交、新依赖、配置改动、环境差异

4. **在多组件系统里收集证据**

   **当系统有多个组件时(CI → 构建 → 签名,API → 服务 → 数据库):**

   **提出修复之前,加诊断埋点:**
   ```
   对每个组件边界:
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 核实环境/配置传递
     - 检查每层的状态

   跑一次收集证据,显示【哪里】断了
   再分析证据定位失败组件
   再调查那个具体组件
   ```

   **例(多层系统):**
   ```bash
   # Layer 1: Workflow
   echo "=== Secrets available in workflow: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # Layer 2: Build script
   echo "=== Env vars in build script: ==="
   env | grep IDENTITY || echo "IDENTITY not in environment"

   # Layer 3: Signing script
   echo "=== Keychain state: ==="
   security list-keychains
   security find-identity -v

   # Layer 4: Actual signing
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **这揭示:** 哪一层失败(secrets → workflow ✓,workflow → build ✗)

5. **追踪数据流**

   **当错误深在调用栈里:** 见本目录 `references/root-cause-tracing.md` 的完整反向追踪技术。

   **快速版:**
   - 坏值从哪来?
   - 谁用坏值调了这里?
   - 一直往上追到源头
   - 在源头修,不在症状处修

### 阶段二:模式分析

**修之前先找到模式:**

1. **找可用的例子** —— 在同代码库定位相似的可用代码。和坏掉的相似、却能跑的是什么?
2. **对照参考实现** —— 若在实现某模式,完整读参考实现,别扫读,应用前彻底理解。
3. **找出差异** —— 可用与坏掉之间有什么不同?列出每个差异,再小也列。别假设「那不可能有关系」。
4. **理解依赖** —— 它还需要哪些组件?什么设置/配置/环境?它做了什么假设?

### 阶段三:假设与测试

**科学方法:**

1. **形成单一假设** —— 清楚陈述:「我认为 X 是根因,因为 Y」。写下来。具体,别含糊。
2. **最小化测试** —— 做能测这个假设的【最小】改动。一次一个变量。别同时修多个。
3. **继续前先验证** —— 有用吗?是 → 阶段四。没用 → 形成【新】假设。别在上面叠更多修复。
4. **当你不知道时** —— 说「我不理解 X」。别装懂。求助。再研究。

### 阶段四:实现

**修根因,不修症状:**

1. **建失败测试用例** —— 最简复现;能自动化就自动化;没框架就一次性脚本;修之前必须有;用 `test-driven-development` 技能写像样的失败测试。
2. **实现单一修复** —— 针对已识别的根因;一次一个改动;没有「顺手」改进;没有捆绑重构。
3. **验证修复** —— 现在测试过吗?没弄坏别的测试吗?问题真解决了吗?
4. **若修复没用** —— 停。数一下:你试了几次修复?<3:回阶段一,带新信息重新分析。**≥3:停,质疑架构(下面第 5 步)**。不做架构讨论就别试第 4 次。
5. **若 3+ 次修复失败:质疑架构**

   **指示架构问题的模式:**
   - 每个修复在不同地方揭示新的共享状态/耦合/问题
   - 修复需要「大规模重构」才能落地
   - 每个修复在别处制造新症状

   **停,质疑根本:** 这个模式根本上成立吗?我们是「纯靠惯性硬扛」吗?该重构架构还是继续修症状?

   **在尝试更多修复前和搭档讨论。** 这不是假设失败——这是架构错了。

## 危险信号——停,走流程

若你发现自己在想:
- 「先快速修一下,回头再查」
- 「就改改 X 看能不能行」
- 「加多个改动,跑测试」
- 「跳过测试,我手动验」
- 「大概是 X,我修一下」
- 「我没完全懂但这可能有用」
- 「模式说 X 但我改着用」
- 「主要问题是:[没调查就列修复]」
- 追踪数据流之前就提解法
- **「再试一次修复」(已试 2+ 次)**
- **每个修复在不同地方揭示新问题**

**这些全都意味着:停。回阶段一。**
**若 3+ 次修复失败:** 质疑架构(见阶段 4.5)。

## 搭档「你做错了」的信号

**留意这些纠偏:**
- 「那没发生吗?」—— 你没验证就假设了
- 「它会给我们看……吗?」—— 你本该加证据收集
- 「别猜了」—— 你在没理解的情况下提修复
- 「深想一下」—— 质疑根本,不只是症状
- 「我们卡住了?」(沮丧)—— 你的方法不管用

**看到这些:** 停。回阶段一。

## 常见合理化

| 借口 | 真相 |
|------|------|
| "问题简单,不用流程" | 简单问题也有根因。流程对简单 bug 很快。 |
| "紧急,没空走流程" | 系统化比瞎试快。 |
| "先试这个,再调查" | 第一个修复定了基调。一开始就做对。 |
| "确认修复有效后再写测试" | 没测的修复站不住。先测才证明。 |
| "一次多个修复省时间" | 没法隔离哪个起作用,还制造新 bug。 |
| "参考太长,我改着用模式" | 部分理解保证有 bug。完整读。 |
| "我看到问题了,修它" | 看到症状 ≠ 理解根因。 |
| "再试一次修复"(2+ 次失败后) | 3+ 次失败 = 架构问题。质疑模式,别再修。 |

## 速查

| 阶段 | 关键活动 | 成功标准 |
|------|----------|----------|
| **1. 根因** | 读错误、复现、查改动、收证据 | 理解【什么】与【为什么】 |
| **2. 模式** | 找可用例子、对比 | 找出差异 |
| **3. 假设** | 形成理论、最小测试 | 确认或新假设 |
| **4. 实现** | 建测试、修、验 | bug 解决、测试通过 |

## 当流程揭示「没有根因」

若系统调查确实揭示问题纯属环境/时序/外部:
1. 你已完成流程
2. 记录你调查了什么
3. 实现合适的处理(重试、超时、错误信息)
4. 加监控/日志供将来调查

**但:** 95% 的「没有根因」是调查不彻底。

## 现实影响

来自调试实战:
- 系统化:15-30 分钟修好
- 随机修:2-3 小时乱撞
- 一次修对率:95% vs 40%
- 引入新 bug:近零 vs 常见
