## 优先级分层与任务清单

### 🔴 真 bug(用户可命中)

#### T1 — MCP 冷启动期 dispatch 线程冻结【real-bug / M】
- 现状/根因:`AppServerMcp.enable` 已 offload 到 `mcpControlExecutor`(:150-154),但 `enable`→`McpServerManager.enable`(synchronized,:243)在整个 `client.initialize()`(npx 冷拉,5-30s+)期间持 manager 内置锁。而两条路径**仍在 dispatch 线程内联执行**并争同一把锁:
  1. `mcp.config.upsert/remove`:`handleMcp` 内联 `action.accept(ops)`(AppServer.java:137-149/304-312)→ `configUpsert`→`reloadFromConfig`(synchronized,McpServerManager.java:602);
  2. 同工作区 `session.start`:`handleSessionStart`→`factory.create`(内联,AppServer.java:216)→ `ensureFor`→`reattach`(synchronized,:590)。
  期间 dispatch 单线程阻塞 → 所有后续 RPC 停摆,UI 冻结冷拉时长。
- 修复方向(**设计取舍,需确认**):
  - 方案 A(推荐,锁窄化):把 manager 的快速记账(config 交换 / `servers.put`)与慢 `start()` 拆分,内置锁**不跨** `client.initialize()`。一处修复同时解掉 reloadFromConfig 与 reattach 两个受害点。
  - 方案 B(offload,较浅):把 `configUpsert/configRemove` 的 `reloadFromConfig` 也 offload 到 `mcpControlExecutor`(镜像 disable);但 `reattach` 在 session.start 关键路径需同步换 registry,offload 会引入时序复杂度,故 reattach 仍需锁窄化。
  - 倾向 A。
- 验收:构造慢 enable(fake 冷拉 sleep)在途,并发发 `mcp.config.upsert` 与同工作区 `session.start`,断言二者响应**不被** enable 时长阻塞(响应延迟 << 冷拉时长);红绿:回退到持锁版本则断言超时变红。

#### T2 — 自动化流水线「砖死」(spawn 同步抛)【real-bug / S】
- 现状/根因:`automationRunner.ts:42` `spawn(...)` 在 `run()` executor(:39)内**无 try/catch**。同步抛(非法 arg/options `ERR_INVALID_ARG_*`、fd 耗尽 EMFILE/ENFILE——**非**坏路径,坏路径走异步 `error` 事件已处理)→ executor 抛 → `this.child` 未赋值 → `exit`/`error` 监听未挂 → `exited`(:29/52/59)**永不 resolve**。`automationScheduler.ts` 的 `.finally` 靠 `runner.exited.then(...)` 释放槽(:169-172)→ `this.current` 永不清 → `drainQueue` 永不跑 → 单并发槽永久占用 → 此后所有定时/手动任务静默排队永不触发,仅重启 app 可解。tick 兜底 `if(!this.current...)`(:112)因 current 卡非空而无法救。
- 修复方向:`spawn` 调用(或 executor 体)套 try/catch;catch 内 `this.exitedResolve?.()` 并派发终态 `child-exit`/`turn.failed`,复用异步 `error` 处理器同一 `exited`-resolve 路径释放槽。
- 验收:注入 spawn 同步抛的 fake,断言 run() 终态为 failed **且** exited resolve **且**后续 runNow 能正常触发(槽已释放);红绿:去掉 try/catch 则第二次 runNow 永久 pending。

#### T3 — 提交失败纯静默(无任何 UX)【robustness→用户可见 / S】
- 现状/根因:`App.tsx:348-356` `handleSubmit` catch 仅 `console.error` + 派发合成 `turn.failed` 解卡 running;`handleEditMessage`(:456-462)同样。后端连着但拒绝/报错时:用户气泡已回显(addUserItem 在 await 前)、composer 已清空、turn 静默转 idle,无助手回复无错误无解释。(死后端另有 DisconnectedBanner 部分兜底,但连着报错的子情形纯静默。)现有 setError/banner 基建(McpServerForm/ModelSwitcher/DisconnectedBanner/ModelFallbackBanner)主链未用。
- 修复方向:`handleSubmit`(及 `handleEditMessage`)catch 增错误提示——复用横幅/瞬态错误态模式(镜像 ModelFallbackBanner),testid 便于 E2E。
- 验收:mock 令 turn.submit 报错,E2E 断言用户可见错误提示出现;红绿:去掉提示则断言找不到元素。

### 🟡 真但边缘 / 低概率

#### T4 — B5 结算窗:同任务 runNow ≤2s 静默无效【robustness / M】
- 现状/根因:B5 严格并发1(`4b3790f`)刻意持槽到子进程真退(`exited.then` 清 current,可滞后结算达 `SIGKILL_UPGRADE_MS=2000ms`)。窗内**同任务**手动 runNow→`activeTaskIds()` 仍含该 task→返回 `{ok:false}`(automationScheduler.ts:64);渲染层 `handleRunNow`(AutomationsPanel.tsx:52-55)**忽略返回值**仅 `setTab('runs')` → 静默无反应。(不同任务走排队 ok:true,正常;调度 tick 落窗记 miss,~6.7%/边界且可见记录为 miss,非全静默。)
- 修复方向:`handleRunNow` surface `ok:false`(toast「任务正在收尾,稍后重试」);或区分「结算中」与「运行中」使同任务手动重试排队而非硬 false。tick-miss:current 的 run 行进入终态后从 `activeTaskIds()` 剔除。
- 验收:E2E/单测构造结算窗内同任务 runNow,断言用户得到「收尾中」反馈(而非静默);红绿:忽略返回值则无反馈。

#### T5 — tools/list_changed 回调无锁 replaceTools【robustness / S】
- 现状/根因:`McpServerManager.java:517-525` 回调(reader 线程)`buildToolList`+`replaceTools`+`server.tools` 无 `synchronized`;`finalizeReadyLocked`(:479-487)做同样操作却持锁与 reattach 互斥。真实危害:list_changed 与 reattach 并发时对旧/新 registry 的丢更新窗口(自愈,无内存损坏)。
- 修复方向:回调体套 `synchronized(this)`(或走小型加锁 helper)与 reattach/finalizeReadyLocked 互斥;注意回调在 reader 线程当前不持任何 manager 锁,加锁安全无嵌套锁反转。
- 验收:补单测断言 list_changed 与 reattach 串行(可注入 latch 观测);或至少加锁后回归绿 + 锁不变量注释。

#### T6 — DiffView 空盒(no-op 写)【cosmetic / S】
- 现状/根因:`ToolRegistry.java:305-321` `write_file` 读旧内容入 `before`,**无 before===content 短路**即触发 `writeFileObserver`(:321)→ 前端 `DiffView` 收到 before===after → `hideUnchangedRegions` 全折叠 + `+0/-0` → 小空盒。LLM 用相同内容重写文件即达(不常见但可达,「生产近不可达」偏乐观)。
- 修复方向:(a)`ToolRegistry` 在触发 observer 前 `before===content` 短路(同时净化 TUI/inline);或(b)`DiffCard`/`DiffView` 在 before===after / 无 line change 时渲染「无变更」占位。倾向 (a)(根治,一处修多端)。
- 验收:单测/E2E 断言 no-op 写不产生空 diff 卡(或产生「无变更」占位);红绿:回退无短路则空盒复现。

### ⚪ 廉价硬化 / 测试补齐 / 文案

#### T7 — `McpServerManager.close()` 加 synchronized【robustness / S】
- 现状:`close()`(:667-674)无 synchronized;当前调用方(`ensureFor` synchronized + `shutdownNow` 先行 + `closed` volatile 各入口再校验)已串行化,无观测危害。
- 修复:为对称与未来调用方兜底,`close()` 加 `synchronized`。验收:回归绿 + 与其他生命周期方法一致。

#### T8 — mcp.status ERROR 通知链端到端断言【test-gap / S】
- 现状:生产链正确(`setStatus(ERROR)`→listener→`pushStatus`→`emitMcpStatus(error)`),但无单一测试贯穿;`SlowFakeManager` 绕过真 `setStatus` 无法覆盖。
- 修复:加 `McpServerManagerTest` 用例——挂录制 `setStatusListener`,启一个必失败 server(复用 `singleServerFailureDoesNotBlockOthers` 的 bad/未设 `${VAR}` 模式),断言收到 ERROR 转换 + 非空 errorMessage。验收:新测试绿 + 故意断链变红。

#### T9 — CodeIndex 部分索引失败在最终 summary 反映【robustness / S】
- 现状:per-file catch(:97-101)可见 emit+warn(**非静默**),但失败文件被丢出 `entries` 后 `index()` 仍返回「索引完成」成功态(:111-113)→ 抖动后语料部分索引却报成功。
- 修复:循环内计 `failedFiles`,反映进 `IndexResult.message`(如「索引完成:N 块(M 个文件失败,建议重试)」)。验收:注入部分失败,断言 summary 含失败计数。

#### T10 — stderr 多行 chunk 逐行前缀【cosmetic / S】
- 现状:`automationRunner.ts:49` 整 chunk 前缀一次,多行仅首行带 taskId(stdout 已 readline 逐行,stderr 未)。
- 修复:stderr 走 readline 逐行(或 `\n` 拆分)逐行前缀。验收:多行 stderr 每行带前缀断言。

### 🔵 潜在守卫(防御性,非当前 bug——用户明确要求纳入)

#### T11 — interrupt turnId 硬化【hardening / S,无当前影响】
- 定性:当前后端按线程中断、不读 turnId,单线程 serve 保证正确落点 → **无实际 bug**。仅为「未来若后端改按 turnId 中断」预防。
- 修复:渲染层从 `submitTurn` resolved 结果捕获 turnId,或 main 侧优雅兜底(early 窗不发 stale/null)。**明确标注为硬化,不改变当前行为**。验收:单测断言 early 窗 interrupt 携带正确/无害 turnId;不追求行为可见变化。

#### T12 — 会话级通知过滤【latent / M,为多会话预埋】
- 定性:app 严格单会话 v1,当前无串台可能(automation 走独立 `onAutomationEvent` 通道)。后端通知已携带 `sessionId`(AppServer.java:246/254)但前端未过滤(main/index.ts:139-141 verbatim 转发;renderer onEvent 不查 sessionId)。
- 修复:在 `sendEvent`/`onEvent` 边界按活跃 `sessionId` 过滤/路由不匹配通知(字段已在,加过滤非改协议)。**明确标注:为多会话预埋,单会话下行为不变**。验收:单测注入异 sessionId 通知,断言被丢弃/不入活跃会话 state;单会话现有流零回归。
