# 完成前的验证(Verification Before Completion)

## 概述

未经验证就声称完成,是不诚实,不是效率。

**核心原则:证据先于断言,永远。**

**违反本规则的字面,就是违反本规则的精神。**

## 铁律

```
没有「新鲜的验证证据」,就不能下「完成」结论
```

如果你没有在这条消息里跑过验证命令,你就不能声称它通过。

## 关卡函数

```
在声称任何状态、或表达满意之前:

1. 识别:哪条命令能证明这个结论?
2. 运行:执行【完整】命令(全新、完整)
3. 阅读:读全部输出,查退出码,数失败数
4. 核实:输出是否证实了结论?
   - 若否:带证据说明真实状态
   - 若是:带证据下结论
5. 才:下结论

跳过任何一步 = 撒谎,不是验证
```

## 常见翻车

| 结论 | 需要 | 不够 |
|------|------|------|
| 测试通过 | 测试命令输出:0 失败 | 上次运行、"应该过" |
| Linter 干净 | Linter 输出:0 错误 | 局部检查、外推 |
| 构建成功 | 构建命令:exit 0 | Linter 过了、日志看着行 |
| Bug 修好 | 测原始症状:通过 | 改了代码、以为修好 |
| 回归测试有效 | 红-绿循环已验证 | 测试过了一次 |
| Agent 完成了 | 版本控制 diff 显示改动 | Agent 报告"成功" |
| 需求满足 | 逐行对照清单 | 测试通过 |

## 危险信号——停

- 用「应该」「大概」「看起来」
- 验证前就表达满意(「太好了!」「完美!」「搞定!」等)
- 没验证就要 commit/push/PR
- 轻信 agent 的成功报告
- 依赖局部验证
- 想「就这一次」
- 累了、想赶紧收工
- **任何「没跑验证却暗示成功」的措辞**

## 反合理化

| 借口 | 真相 |
|------|------|
| "现在应该行了" | 去跑验证 |
| "我很有信心" | 信心 ≠ 证据 |
| "就这一次" | 无例外 |
| "Linter 过了" | Linter ≠ 编译器 |
| "Agent 说成功了" | 独立验证 |
| "我累了" | 疲惫 ≠ 借口 |
| "局部检查够了" | 局部证明不了什么 |
| "换个说法规则就不适用了" | 精神高于字面 |

## 关键范式

**测试:**
```
✅ [跑测试命令] [看到:34/34 通过] "全部测试通过"
❌ "现在应该过了" / "看着对"
```

**回归测试(TDD 红-绿):**
```
✅ 写 → 跑(过)→ 撤销修复 → 跑(必须失败)→ 恢复 → 跑(过)
❌ "我写了个回归测试"(没做红-绿验证)
```

**构建:**
```
✅ [跑构建] [看到:exit 0] "构建通过"
❌ "Linter 过了"(linter 不检查编译)
```

**需求:**
```
✅ 重读计划 → 建清单 → 逐条核 → 报告缺口或完成
❌ "测试过了,阶段完成"
```

**Agent 委派:**
```
✅ Agent 报成功 → 查版本控制 diff → 核实改动 → 报告真实状态
❌ 信 agent 报告
```

## 为什么重要

来自 24 条失败记忆:
- 搭档说「我不信你」——信任崩了
- 发出去的函数没定义——会崩
- 发出去缺需求——功能不完整
- 因假完成浪费时间 → 被打回 → 返工
- 违反:「诚实是核心价值。撒谎会被替换。」

## 何时适用

**总是,在以下之前:**
- 任何形式的成功/完成结论
- 任何表达满意
- 任何关于工作状态的正面陈述
- commit、建 PR、任务完成
- 转下一个任务
- 委派给 agent

**规则适用于:**
- 原话
- 转述与同义
- 成功的暗示
- 任何暗示完成/正确的表达

## wraith 专项验证锚点

- 桌面(TS):`typecheck` / `npx vitest run`;后端(Java):`mvn compile` / `mvn package`。
- **改了 Java 后端却「桌面没变化」→ 九成是没重打 jar / 没重启**,不是修好了。
- 密钥红线:`git diff --cached | grep -iE 'api[_-]?key|secret|sk-|Bearer'`。

## 底线

**验证没有捷径。** 跑命令,读输出,然后再下结论。不可谈判。
