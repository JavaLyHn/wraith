# systematic-debugging Skill

## 核心原则

**每一步都靠证据,不靠猜。** 乱改一通碰运气,比系统排查更慢、还会引入新问题。

## 五步法

1. **复现**:找到稳定触发的最小路径。复现不了就先让它能复现(加日志/缩小输入)。不能复现 = 不能确认修好。
2. **定位**:用证据把范围一层层缩小——读报错栈、加日志、二分注释、对比「能跑 vs 不能跑」的差异。缩到具体那几行。
3. **根因**:问「为什么会这样」直到触底。表象(报错点)常常不是根因(见 [[root-cause-tracing]])。改表象 = bug 会换个姿势回来。
4. **最小修复**:针对根因做最小改动。别顺手重构、别一次修多个可能。
5. **验证**:复现路径不再触发 + 跑相关测试 + 想清楚有没有副作用/回归。

## 关键纪律

- **一次只改一个变量**:同时改多处,修好了也不知道是哪条起的作用。
- **不要在同一处反复重试**:第一次失败要读懂失败信息(它是证据),据此换方向,而不是原地重跑。
- **区分「症状」和「根因」**:`NullPointerException` 在 A 行,根因可能在 B 处没初始化。
- **改完能解释**:说得出「原来为什么错、这个改动为什么对」,才算修好;说不出就是还没到根因。

## 与 wraith 的衔接

- 有 snapshot:大改动前后可回滚,放心排查。
- 改完按项目铁律验证:相关 `vitest` / `mvn` / typecheck;涉及后端还要重打 jar 才生效。

## 反模式

- ❌ 没复现就宣称修好。
- ❌ 看到报错行就地打补丁,不追根因。
- ❌ 一把梭改五个地方 + 重跑碰运气。
- ❌ 把「不报错了」当「修好了」——可能只是掩盖了症状。
