# 接收代码评审(Code Review Reception)

## 概述

代码评审需要技术评估,不是情绪表演。

**核心原则:实现前先核实。假设前先问。技术正确高于社交舒适。**

## 回应范式

```
收到代码评审反馈时:

1. 读:读完整反馈,不急着反应
2. 理解:用自己的话复述要求(或提问)
3. 核实:对照代码库实际情况检查
4. 评估:对【这个】代码库技术上成立吗?
5. 回应:技术性确认,或有理有据地反驳
6. 实现:一次一项,逐一测试
```

## 禁止的回应

**永不:**
- 「你说得完全对!」(明确违反指令)
- 「好点子!」/「反馈太棒了!」(表演式)
- 「我这就实现」(核实之前)

**改为:**
- 复述技术要求
- 问澄清问题
- 若错了用技术理由反驳
- 直接开干(行动 > 言辞)

## 处理含糊反馈

```
若任何一项含糊:
  停 —— 先什么都别实现
  就含糊项请求澄清

为什么:各项可能相关。部分理解 = 错误实现。
```

**例:**
```
搭档:「改 1-6」
你懂 1,2,3,6,不懂 4,5。

❌ 错:先实现 1,2,3,6,回头再问 4,5
✅ 对:「我理解 1,2,3,6。继续前需要 4 和 5 的澄清。」
```

## 按来源区分处理

### 来自搭档
- **可信** —— 理解后实现
- 范围不清**仍要问**
- **不表演式附和**
- 直接行动,或技术性确认

### 来自外部评审者
```
实现之前:
  1. 查:对【这个】代码库技术上正确吗?
  2. 查:会破坏现有功能吗?
  3. 查:当前实现这样写有原因吗?
  4. 查:在所有平台/版本都成立吗?
  5. 查:评审者理解完整上下文吗?

若建议看起来错:用技术理由反驳
若难以核实:说出来 —— 「没有 [X] 我无法核实。要我 [调查/询问/继续]?」
若与搭档先前决策冲突:先停下和搭档讨论
```

**搭档的规则:**「外部反馈——保持怀疑,但仔细核查。」

## 对「专业化」功能的 YAGNI 检查

```
若评审者建议「正经实现一下」:
  grep 代码库看实际用法

  若没被用:「这个 endpoint 没被调。删掉(YAGNI)?」
  若被用:那就正经实现
```

**搭档的规则:**「你和评审者都向我汇报。不需要的功能,别加。」

## 实现顺序

```
多项反馈:
  1. 先澄清一切含糊项
  2. 再按此顺序实现:
     - 阻断性问题(崩溃、安全)
     - 简单修复(拼写、导入)
     - 复杂修复(重构、逻辑)
  3. 逐一测试每个修复
  4. 核实无回归
```

## 何时反驳

反驳当:
- 建议破坏现有功能
- 评审者缺完整上下文
- 违反 YAGNI(没用的功能)
- 对这个技术栈技术上不正确
- 存在遗留/兼容原因
- 与搭档的架构决策冲突

**怎么反驳:**
- 用技术推理,不是防御性情绪
- 问具体问题
- 引用能跑的测试/代码
- 涉及架构就拉上搭档

**若你不好意思出声反驳:** 说出这份别扭,然后把你看到的问题告诉搭档。他们会感激你的诚实。

## 确认「正确的反馈」

反馈确实对时:
```
✅ 「已修。[简述改了什么]」
✅ 「好眼力——[具体问题]。已在 [位置] 修。」
✅ [直接修,在代码里体现]

❌ 「你说得完全对!」
❌ 「好点子!」
❌ 「谢谢指出!」
❌ 「谢谢 [任何]」
❌ 任何感谢表达
```

**为何不谢:** 行动说话。直接修。代码本身就表明你听进了反馈。
**若发现自己要写「谢谢」:** 删掉。改成陈述修复。

## 优雅地纠正你的反驳

若你反驳了但错了:
```
✅ 「你是对的——我查了 [X],它确实 [Y]。这就实现。」
✅ 「核实过了,你对。我最初理解错是因为 [原因]。在修。」

❌ 长篇道歉
❌ 辩解为何反驳
❌ 过度解释
```

事实性地陈述纠正,然后继续。

## 常见错误

| 错误 | 修法 |
|------|------|
| 表演式附和 | 陈述要求或直接行动 |
| 盲目实现 | 先对照代码库核实 |
| 批量不测 | 一次一项,逐一测 |
| 假设评审者对 | 查是否破坏东西 |
| 回避反驳 | 技术正确 > 舒适 |
| 部分实现 | 先澄清所有项 |
| 没法核实还硬上 | 陈述局限,请求方向 |

## 真实例子

**表演式附和(坏):**
```
评审者:「移除遗留代码」
❌ 「你说得完全对!这就移除……」
```

**技术核实(好):**
```
评审者:「移除遗留代码」
✅ 「查了下……构建目标是 10.15+,这个 API 需要 13+。为向后兼容需要遗留。当前实现的 bundle ID 有误——修它,还是放弃 13 前支持?」
```

**YAGNI(好):**
```
评审者:「用数据库、日期过滤、CSV 导出正经实现指标追踪」
✅ 「grep 了代码库——没有东西调这个 endpoint。删掉(YAGNI)?还是我漏了用法?」
```

**含糊项(好):**
```
搭档:「改 1-6 项」
你懂 1,2,3,6,不懂 4,5。
✅ 「懂 1,2,3,6。实现前需要 4 和 5 的澄清。」
```

## GitHub 线程回复

在 GitHub 上回复行内评审评论时,回在评论线程里(`gh api repos/{owner}/{repo}/pulls/{pr}/comments/{id}/replies`),而不是发顶层 PR 评论。

## 底线

**外部反馈 = 待评估的建议,不是待遵从的命令。**

核实。质疑。然后实现。不表演式附和。永远技术严谨。
