# test-driven-development Skill

## 核心循环:红 → 绿 → 重构

1. **红**:先写测试,描述「对的行为该是什么」,运行它,**看它失败**(证明测试真的在测东西)。
2. **绿**:写**最少**的实现让测试通过。先求对,不求漂亮。
3. **重构**:测试保护下清理实现/命名/重复。每次重构后重跑测试保持绿。

## 什么时候用 TDD

- **纯逻辑函数**(输入→输出确定):最适合,先写测试收益最大。
- **修 bug**:先写一个能**复现 bug 的失败测试**,再修到它变绿——顺带拿到回归防护。
- **不强求**:纯 UI 视觉、一次性脚本、探索性原型——可后补或免测。

## 关键纪律

- **先看红**:没见过失败的测试不可信(可能永远绿、根本没断言到)。
- **测行为,不测实现**:断言「做了什么」,而非「怎么做的」,否则重构一动测试就碎。
- **一次一个行为**:小步快跑,别一个测试塞十个断言。
- **边界优先**:空/零/负/超长/并发/异常路径往往才是 bug 藏身处。

## 与 wraith 的衔接

- 桌面(TS):纯函数抽到 `lib/`,配 `desktop/test/*.test.ts`(vitest),`npx vitest run <名>`。
- 后端(Java):JUnit;注意项目测试默认跳过,需 `-DskipTests=false`,且有已知 JDK/Mockito 噪声基线。
- **测试隔离铁律**:测试绝不写真实 `~/.wraith/config.json` / 索引 / 审计 / memory;只测纯逻辑或用临时路径。

## 反模式

- ❌ 先写实现再补测试(那不是 TDD,且容易「照着实现写测试」自我循环)。
- ❌ 测试从没失败过就当它有效。
- ❌ 把实现细节焊进断言,重构即碎。
- ❌ 为凑覆盖率写无意义断言。
