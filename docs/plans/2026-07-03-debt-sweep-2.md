### Task 1: MCP 冷启动 dispatch 冻结——锁窄化(方案 A)
- 目标:manager 内置锁**不跨** `client.initialize()` 冷拉,使 `reloadFromConfig`(config.upsert/remove)与 `reattach`(同工作区 session.start)不被在途慢 enable 阻塞。
- 现状/根因:见已钉事实。`enable`(synchronized:243)→`start`→`client.initialize()`(:445 前后)整段持锁;`reloadFromConfig`(:602)/`reattach`(:590)争同锁,而其调用方在 dispatch 单线程内联 → 冷拉期全 RPC 停摆。
- 改动点(McpServerManager.java):把慢 `start()` 的 `client.initialize()` 移出内置锁——锁内只做快速记账(状态置 STARTING、servers 记录、closed 校验),`initialize()` 在锁外执行,完成后再入锁做 `finalizeReadyLocked`(已存在的加锁定稿点)。确保 `reloadFromConfig`/`reattach` 只需短暂持锁做 registry 交换,不等冷拉。保持 `closed` 各入口再校验语义。
- 验收(红绿):新单测——注入一个 `initialize()` 阻塞可控(latch/sleep)的 fake client 触发慢 enable;并发调 `reloadFromConfig` 与 `reattach`,断言二者在 enable 完成前即返回(不被阻塞)。红:回退到 initialize 持锁版,断言超时/被阻塞变红。
- 门禁:Java 全量绿 + 新并发单测;`AppServerMcpTest`/`McpServerManagerTest` 既有用例零回归。

### Task 2: 自动化流水线砖死——spawn 同步抛兜底
