# code-review Skill

## 核心原则

**带着维度系统扫,而不是从头读到尾凭感觉。** 每条意见要能定位到行、说清「为什么是问题」、给出改法。

## 审查维度(逐个过)

1. **正确性**:逻辑对不对?边界(空/零/负/越界/并发)?错误路径处理了吗?返回值/异常?
2. **安全**:注入(SQL/命令/路径穿越)、鉴权、**密钥泄漏**(日志/回包/前端)、越权、不可信输入。
3. **可维护性**:命名、重复、复杂度、和现有代码风格是否一致、注释密度是否匹配周围。
4. **测试**:关键逻辑有测试吗?边界覆盖吗?测试是否测行为而非实现?
5. **契合度**:是否引入不必要依赖/抽象?是否符合本仓约定(AGENTS.md/CLAUDE.md)?

## 意见怎么给

- **定位到行** + **严重度**标注:🔴 必改(bug/安全) / 🟡 建议 / 🟢 可选/风格。
- 说清**为什么**是问题 + **怎么改**(给方向或代码)。
- **先看有没有 bug,再谈风格**:别让 nitpick 淹没真问题。
- 客观、对事不对人;拿不准的标「疑问」而非「错误」。

## 与 wraith 的衔接

- 本仓专项红线务必查:异常只报 `e.getClass().getSimpleName()`;密钥只进 `~/.wraith/config.json`,不进日志/RPC 回包/renderer;测试不碰真实 config/索引/审计/memory。
- 大改动可要求配套验证证据(编译/测试/构建),见 [[verification-before-completion]]。

## 反模式

- ❌ 只挑格式/命名,放过逻辑与安全问题。
- ❌ 意见含糊(「这里怪怪的」)——没定位、没改法。
- ❌ 所有意见一个严重度,让人分不清轻重。
- ❌ 把个人风格偏好当硬性错误。

> 方法论改编自社区 code-review 实践(obra/superpowers 等)。
