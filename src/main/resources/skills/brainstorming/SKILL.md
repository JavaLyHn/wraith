# brainstorming Skill

## 核心原则

**需求不清就不写代码。先把「解空间」摊开,和用户对齐一个方案,再动手。**

跳过发散直接实现,是返工的头号来源。这个技能的目的:用几分钟对齐,省几小时重写。

## 什么时候用

- 用户抛来一个**新功能/新需求/开放问题**,而「怎么做」有多种合理走法。
- 需求描述模糊、含隐含假设、或你脑子里已经冒出「其实有两三种做法」。
- **不该用**:任务已经很明确、只有一条显然路径、或纯机械改动——那种直接做。

## 流程

1. **澄清目标**:先复述你对需求的理解 + 成功标准。有真正影响方案的未知,**问**(用最少的关键问题,别查代码能查到的)。
2. **发散 ≥2 个方案**:给出 2–3 个**实质不同**的走法,不是同一方案的措辞变体。每个方案写:
   - 一句话思路
   - **取舍**(成本/风险/可维护性/契合现有架构的程度)
3. **给推荐**:明确说「我建议 A,因为…」,而不是把选择全甩给用户。第一顺位放推荐方案。
4. **对齐再动手**:等用户拍板(或用 AskUserQuestion 让其选)。方案定了,再考虑是否进 `/plan` 或直接实现。

## 与 wraith 工作流的衔接

- 方案定了之后:复杂任务 → 写 spec 文档(见 [[writing-plans]] / 项目 spec-first 约定)再实现;简单任务直接做。
- 实现走 TDD 的,见 [[test-driven-development]]。

## 反模式

- ❌ 只给一个方案就开写(没得选 = 没 brainstorm)。
- ❌ 罗列一堆方案却不给推荐(把认知负担全推给用户)。
- ❌ 为了「显得全面」发散五六个雷同选项。
- ❌ 明明任务很清楚还强行发散,拖慢节奏。
