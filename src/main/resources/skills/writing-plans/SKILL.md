# writing-plans Skill

## 核心原则

**先写清楚「做什么、怎么验证」,再写代码。** 一份好计划让实现变成「照着做」,而不是「边做边想」。

## 什么时候写计划

- 任务**跨多个文件/多个步骤**,或改动面不小。
- 有明确的验收标准需要事先钉死。
- **不用写**:单点小改、一目了然的修复——直接做 + 验证即可。

## 一份合格计划包含

1. **目标 & 非目标**:要解决什么;明确**不做**什么(防范围蔓延)。
2. **现状事实**:相关文件/接口/约束(基于真实代码,不臆测——不确定就先查)。
3. **步骤**:有序、每步足够小可独立验证;标出步骤间依赖。
4. **验证方式**:每步/整体怎么证明对了(编译、测试、手动核验点)。
5. **风险/回滚**:已知风险 + 出错怎么退(wraith 有 snapshot 可兜底)。

## 与 wraith 工作流的衔接

- 项目约定 **spec-first**:重要功能先在 `docs/superpowers/specs/<date>-<name>.md` 落一份 spec 再实现。
- 计划里的「验证方式」应包含项目铁律:`typecheck` / `vitest` / `mvn 编译` / 密钥红线扫描。
- 方案本身不明时,先 [[brainstorming]] 对齐走法,再写计划。

## 反模式

- ❌ 计划全是「实现 X 功能」这种没法执行的空话——步骤要具体到文件/动作。
- ❌ 只列步骤不列验证——做完不知道对没对。
- ❌ 基于臆测写计划——先读代码确认现状再写。
- ❌ 计划写完就锁死——执行中发现前提错了,回来改计划,别硬扛。
