# verification-before-completion Skill

## 核心原则

**「应该可以了」不是完成,「我验证过了」才是。** 未经验证就报完成,是信任的头号杀手。

## 完成前检查清单

宣称完成前,逐项确认(适用者):

- [ ] **编译/类型通过**:`typecheck` / `mvn compile` 无错。
- [ ] **测试通过**:相关 `vitest` / JUnit 真的跑了且绿——**看到输出**,不是「应该绿」。
- [ ] **构建通过**:涉及产物的跑 `build` / `mvn package`。
- [ ] **实际行为对**:改的功能亲自触发一遍(或说明为何无法本地验证)。
- [ ] **无回归**:想清楚这次改动的副作用面,必要时跑全量测试。
- [ ] **项目专项铁律**:密钥红线扫描(`git diff --cached | grep -iE 'api[_-]?key|secret|sk-|Bearer'`);改了 Java 记得**重打 jar + 同步 ~/.wraith/wraith.jar + 重启桌面**才生效。

## 如实报告

- 验证通过 → 直接说「已完成并验证:<怎么验的>」。
- **有没做/没验的** → 明说,别粉饰:「X 已改但没跑真机」「测试 4 个失败是预存基线」。
- 做不到本地验证 → 说明原因 + 给出用户可自验的步骤。

## 与 wraith 的衔接

- 后端改动「看着对」但桌面没生效,九成是**没重打 jar/没重启**(preload 也不热更),不是代码 bug。
- 报「已修复」前,若可能,给一条用户能自己复现验证的操作。

## 反模式

- ❌ 没跑测试就说「测试应该过」。
- ❌ 把「不报错了」当「功能对了」。
- ❌ 完成报告里隐去跳过的步骤/已知问题。
- ❌ 改完后端不重打 jar 就让用户去验(必然「没变化」)。
